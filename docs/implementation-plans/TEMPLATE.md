# 実装計画書テンプレート

状態: 下書き
作成日: YYYY-MM-DD
作成者: <name>

## 1. 目的

この変更で何を達成したいかを 2〜5 行で書く。

## 2. 背景

- なぜこの変更が必要か
- いま何が問題か
- 参照すべき既存仕様や不具合票があれば書く

## 3. 変更対象

- `path/to/file-a`
- `path/to/file-b`

## 4. 影響範囲

- 呼び出し元
- 呼び出し先
- 関連設定
- 関連テスト

## 5. 非対象範囲

今回やらないことを明示する。

- 例: 広域リファクタ
- 例: UI 文言変更
- 例: 別モジュールの整理

## 6. 実装方針

実装の考え方を短く書く。

- どのロジックをどう変えるか
- 既存との互換性をどう守るか
- リスクをどう避けるか

## 7. 実装手順

1. 何を変更するか
2. 次に何を変更するか
3. 最後に何を確認するか

## 8. 必須確認項目

- 対象ファイル
- 直近の呼び出し元または依存先
- 関連テスト
- 必要な設定ファイル

## 9. 必須 checks

`loop run` はこのセクションのコマンドを自動実行する。

- `path/to/check-command-1`
- `path/to/check-command-2`

## 10. 受け入れ条件

- 何を満たせば完了か
- どうなっていれば `Code Reviewer` が `approve` できるか

## 11. エスカレーション条件

次のような場合は `replan` または `human` に戻す。

- 計画外の変更が必要になった
- 想定より影響範囲が広い
- セキュリティ、移行、データ更新が必要
- 必要な前提情報が欠けている

## 12. 実装役向けメモ

`Implementer` に特に守らせたいことを書く。

- 例: この計画の範囲外の最適化はしない
- 例: 必ず既存テストを先に読んでから直す
